# アーキテクチャ決定記録 (Architecture Decision Records)

本プロジェクトにおける主要な技術的決定とその背景を記録します。

## 1. 4システムによるモジュール構成の採用
- **決定:** システムを「社員管理」「製品管理」「作業実績管理」「原価計算・集計」の4つに分割。
- **背景:** 
    - 業務領域（ドメイン）ごとに責任を明確に分けることで、開発・保守の複雑さを軽減するため。
    - 将来的に特定のモジュールのみを刷新したり、異なる技術スタックで再構築したりすることを容易にするため。

## 2. データベースに SQLite を採用
- **決定:** 開発およびプロトタイプ段階では SQLite を使用。
- **背景:**
    - 設定不要で即座に開発を開始でき、データベースファイルを含めた可搬性が高いため。
    - 初期段階のデータ量であればパフォーマンス上十分であり、運用のシンプルさを優先した。

## 3. CSVファイルによるシステム間連携
- **決定:** 各モジュール間のデータ連携に共有DBやAPIではなく、CSVファイルを採用。
- **背景:**
    - システム間を疎結合に保つため。
    - データの流れが可視化されやすく、トラブルシューティングが容易。
    - 既存の表計算ソフト（Excel等）でのデータ確認・修正との親和性が高いため。

## 4. 時給データを「社員管理システム」で保持
- **決定:** 社員の時給情報を「原価計算」ではなく「社員管理」側で管理。
- **背景:**
    - 時給は社員の属性（雇用条件）の一部であり、社員マスタの一部として管理するのが自然であるため。
    - 昇給等の履歴管理を社員情報のライフサイクルと同期させるため。

## 5. 製品単価を「製品管理システム」で保持
- **決定:** 製品の単価（材料費・外注費等）を「製品管理」側で管理。
- **背景:**
    - 単価は製品の属性（仕様・コスト構成）の一部であり、製品マスタで一元管理すべきであるため。
    - 製品の改廃と価格改定を同じコンテキストで扱うため。

## 6. PostgreSQL への移行条件
- **決定:** 以下の条件のいずれかに該当する場合、PostgreSQL への移行を実施する。
- **条件:**
    - **同時実行性能:** 複数ユーザーによる同時書き込み頻度が高まり、SQLiteのロック待ち（Database is locked）が頻発する場合。
    - **データ量:** 作業実績データが数万〜数十万件を超え、複雑な集計クエリのパフォーマンスが低下した場合。
    - **機能要件:** 高度な分析関数（ウィンドウ関数等）や、より厳格なアクセス制御・トランザクション管理が必要となった場合。
